# Hyprland Rice Reevaluation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix mako's hanging-notification problem, repair silently broken wallpaper switching, wire ironbar into theming and retire waybar, replace ranger with yazi, and convert the rice into a clean GNU Stow + git dotfiles repo.

**Architecture:** This is a config/dotfiles project — there is NO test framework. "Tests" are manual verification commands with expected observable outcomes (notifications dismissing, themes restyling, symlinks resolving). The repo uses symlinks that currently point *backwards* (repo → `~/.config`); Phase 4 flips them to Stow's convention (files in repo, symlinks out). Several live configs are **generated by flavours** from mustache templates — structural fixes must go into the template AND the live file, or the next theme switch reverts them.

**Tech Stack:** mako, ironbar (GTK4), yazi, flavours (Base16 mustache templating), awww (swww fork), GNU Stow, git, Hyprland.

**Commit policy:** git does not exist until Phase 4. Phases 1–3.5 checkpoint via *verification*, not commits. Phase 4 does `git init` and makes the first baseline commit containing all prior work. After Phase 4, each task commits normally.

**User rules in effect:** Never run sudo/pacman/yay — copy install commands to clipboard with `wl-copy` and tell the user to run them. `stow` is NOT installed; this is handled in Phase 4 Task 1.

Create `~/.config/flavours/templates/ironbar/templates/default.mustache` defining CSS color variables. GTK CSS does not support `@define-color` the way GTK3 apps expect in all cases — ironbar (GTK4) supports standard CSS custom properties via `@define-color` in its stylesheet. Produce a colors file:
```css
/* Base16 {{scheme-name}} — ironbar colors (generated by flavours) */
@define-color base00 #{{base00-hex}};
@define-color base01 #{{base01-hex}};
@define-color base02 #{{base02-hex}};
@define-color base03 #{{base03-hex}};
@define-color base04 #{{base04-hex}};
@define-color base05 #{{base05-hex}};
@define-color base08 #{{base08-hex}};
@define-color base0A #{{base0A-hex}};
@define-color base0B #{{base0B-hex}};
@define-color base0C #{{base0C-hex}};
@define-color base0D #{{base0D-hex}};
@define-color base0E #{{base0E-hex}};
```

- [ ] **Step 3: Register the template in flavours config.toml**

In `~/.config/flavours/config.toml`, ADD a new item (place near the old waybar block):
```toml

[[items]]
file = "~/.config/ironbar/colors.css"
template = "ironbar"
subtemplate = "default"
rewrite = true
hook = "touch ~/.config/ironbar/style.css"
```
(The `touch` hook triggers ironbar's live CSS reload. Leave the waybar item in place for now — it is removed in Task 7.)

- [ ] **Step 4: Make style.css consume the generated colors**

At the TOP of `~/.config/ironbar/style.css`, add:
```css
@import url("colors.css");
```
Then replace the hardcoded Nord hex values found in Step 1 with `@theme01` references — actually GTK4 uses the `@define-color name` then `color: @name;` syntax. Replace e.g. `background-color: #2e3440;` → `background-color: @base00;`, `color: #d8dee9;` → `color: @base05;`, accent borders → `@base0D` / `@base0E`. Map each Nord hex to its Base16 slot by role (darkest bg→base00, fg→base05, blue accent→base0D, purple→base0E, red→base08, green→base0B, etc.).
